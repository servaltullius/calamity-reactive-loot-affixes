## 1) 현재 구조 진단 (2026-02-26 재측정)

### 관찰된 결합도 핫스팟
- `EventBridge` 단일 클래스가 너무 많은 역할을 보유
  - 설정 로드/파싱 (`EventBridge.Config.cpp`)

  - UI/룬워드/루팅/직렬화

  - `EventBridge.Config.*`: 총 3,443 LOC
    - 최대 파일: `EventBridge.Config.RunewordSynthesis.StyleExecution.cpp` (757 LOC)
  - `EventBridge.Loot.Runeword.*`: 총 3,223 LOC
    - 최대 파일: `EventBridge.Loot.Runeword.RecipeEntries.cpp` (638 LOC)
  - `EventBridge.Triggers.Events.cpp`: 795 LOC
- 결합도 재측정(`python3 scripts/vibe.py doctor --full`, 2026-02-26)
  - `EventBridge.h <-> EventBridge.Config.cpp` 변경 결합 상위 유지
  - `EventBridge.h <-> EventBridge.Loot.Assign.cpp` 변경 결합 상위 유지
- 동일 규칙의 중복 구현
  - Player-owned 판정/owner resolve 로직이 `EventBridge`와 `Hooks` 양쪽에 존재

### 리스크
- 규칙이 한쪽만 수정될 때 동작 불일치 가능성 증가
- 회귀 발생 시 원인 추적 범위가 넓어 디버깅 비용 증가
- 신규 기능 추가 시 `EventBridge` 집중도가 더 높아짐

### 기대 효과
- 소유권 판정 규칙 변경 시 수정 지점 단일화
- Hook 경로와 Event 경로 간 판정 일관성 강화
- 전투 판정 정책(소유권/적대관계) 변경 시 호출부 수정 최소화
- first-hit 상태정책 변경 시 EventBridge 비즈니스 코드 영향 축소
- trap 동작 회귀 발생 시 조사 범위를 `Actions.Trap` 모듈로 빠르게 국소화
- cast/adaptive 회귀도 `Actions.Cast` 단일 모듈에서 추적 가능
- 액션 라우팅/가드 회귀를 `Actions.Dispatch` 모듈에서 독립적으로 추적 가능
- 핵심 상태게이트 정책의 회귀를 빌드 단계에서 조기 탐지

### 4.1 시정 항목 A: `Config` 모듈 디커플링 스펙

현 상태(코드 근거):
- 엔트리포인트: `skse/CalamityAffixes/src/EventBridge.Config.cpp::LoadConfig()`
- 구현 모듈: `EventBridge.Config.*` 17개 TU(총 3,443 LOC)
- 문제 핵심: 파싱/검증/인덱싱/합성/적용이 모두 `EventBridge` 내부 상태(`_affixes`, `_loot`, `_runewordRecipes`)를 직접 변이

목표 경계:
- `ConfigParseStage`: JSON -> typed DTO (`ParsedRuntimeConfig`)
- `ConfigBuildStage`: DTO -> 런타임 레지스트리/인덱스 (`BuiltRuntimeConfig`)
- `ConfigApplyStage`: 빌드 결과를 단일 커밋(`ApplyConfigSnapshot`)으로 적용

기술 계약(초안):
- 신규 헤더: `skse/CalamityAffixes/include/CalamityAffixes/RuntimeConfigSnapshot.h`
- 신규 API:
  - `ParseRuntimeConfig(const nlohmann::json&) -> ParseResult<ParsedRuntimeConfig>`
  - `BuildRuntimeConfig(const ParsedRuntimeConfig&, RE::TESDataHandler*) -> BuildResult<BuiltRuntimeConfig>`
  - `ApplyRuntimeConfigSnapshot(const BuiltRuntimeConfig&) -> void`
- 역호환 조건:
  - `RuntimeContract::kSupportedTriggers`, `kSupportedActionTypes` 검증 경로 유지
  - 기존 `LoadConfig()` 외부 시그니처/호출 지점 불변

마이그레이션 단계:
1. Stage 0 (`2026-02-27`): 스냅샷 DTO/계약 헤더 추가, 기존 로직과 병행 가능한 어댑터 작성
2. Stage 1 (`2026-03-01`): `ParseConfiguredAffixesFromJson`, `IndexConfiguredAffixes`, `SynthesizeRunewordRuntimeAffixes`를 빌드 단계 경계로 이동
3. Stage 2 (`2026-03-03`): `LoadConfig()`를 오케스트레이터로 축소(파싱 -> 빌드 -> 커밋), 실패 시 no-op 롤백 보장

완료 기준(DoD):
- `LoadConfig()` 본문 120 LOC 이하
- Config 파이프라인 단위 정적 체크 3종 추가:
  - parse 실패 경계
  - build 단계 계약 위반 경계
  - apply 단계 부분적용 금지(원자적 커밋)
- 기존 커맨드 무회귀:
  - `ctest --test-dir skse/CalamityAffixes/build.linux-clangcl-rel --output-on-failure`
  - `python3 tools/lint_affixes.py --spec affixes/affixes.json --generated Data/SKSE/Plugins/CalamityAffixes/affixes.json`

### 4.2 시정 항목 B: `Loot.Runeword` 모듈 디커플링 스펙

현 상태(코드 근거):
- 구현 모듈: `EventBridge.Loot.Runeword.*` 13개 TU(총 3,223 LOC)
- 문제 핵심: 조회(`RecipeEntries`/`PanelState`)와 명령(`Transmute`/`Reforge`)이 동일 상태를 직접 접근해 읽기/쓰기 경계가 약함
- 결합 지점: `_runewordRecipes`, `_runewordSelectedBaseKey`, `_runewordSelectedRecipeToken`, `_runewordInstanceStates`, `_runeNameByToken`

목표 경계:
- `RunewordQueryService`: UI 조회 전용(`GetRunewordRecipeEntries`, `GetRunewordPanelState`)
- `RunewordCommandService`: 상태 변이 전용(`Select*`, `Transmute`, `Reforge`)
- `RunewordStateStore`: 런타임 상태 + 직렬화 레코드(`kSerializationRecordRunewordState`) 단일 소유

기술 계약(초안):
- 신규 헤더:
  - `skse/CalamityAffixes/include/CalamityAffixes/RunewordStateStore.h`
  - `skse/CalamityAffixes/include/CalamityAffixes/RunewordServices.h`
- 신규 API:
  - `BuildRunewordRecipeEntries(const RunewordReadModel&) -> std::vector<RunewordRecipeEntry>`
  - `ExecuteRunewordTransmute(RunewordCommandContext&) -> OperationResult`
  - `ExecuteRunewordReforge(RunewordCommandContext&) -> OperationResult`
- 역호환 조건:
  - Prisma UI 이벤트 문자열(`CalamityAffixes_Runeword_*`) 유지
  - `RunewordPanelState` DTO 필드 스키마 불변

마이그레이션 단계:
1. Stage 0 (`2026-02-28`): Query/Command 분리 인터페이스 추가, 기존 구현을 어댑터로 연결
2. Stage 1 (`2026-03-02`): `RecipeEntries.cpp`, `PanelState.cpp`를 QueryService로 이관
3. Stage 2 (`2026-03-05`): `Transmute.cpp`, `Reforge.cpp`, `Crafting.cpp`를 CommandService로 이관하고 직렬화 경계 정리

완료 기준(DoD):

- Command 경로에서 UI 텍스트 포맷팅 의존 0건

  - 선택 base/recipe 무효 토큰 정규화
  - transmute/consume 원자성
  - serialize -> load round-trip 동치
